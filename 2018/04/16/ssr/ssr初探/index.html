SSR初探

Posted on 2018-04-16

什么是SSR

SSR是服务端渲染。现在的SPA应用，大多是定义一个根元素，然后动态加载js后，组装根元素，但这样会遇到几个问题：

  • 无法SEO，根元素内容为空，爬虫无法爬到内容
  • 首屏白屏，js加载完成后才会渲染出页面。

而SSR就是为了解决这些问题，将首屏需要的页面结构和数据一起吐给浏览器，即利于SEO，又加速首屏渲染。

项目结构

思路整理

  1. client打包，假设打包后的文件为 bundle.client.js
  2. server打包，假设打包后的文件为 bundle.server.js
  3. server启动，假设为 http://localhost:8080/
  4. 请求页面，假设为 http://localhost:8080/home
  5. server拿到请求url为 /home
  6. server根据 /home 解析出其页面结构和页面所需要的数据，待数据拉取成功后，页面结构放入 <html></html> 中，数据放入哪里待会说，同时将 bundle.client.js 放入 script 标签，一并吐给浏览器。这里有一个问题，到了浏览器，store的数据怎么更新？在server拿到了数据以后，怎么传给浏览器？这里其实通过 <script>window.__INITIAL_STATE__ = ${data}</script> 在window对象上挂载一个属性，存储页面所需要的数据，client只需要判断该属性是否存在，如果存在，替换即可，具体写法是 window.__INITIAL_STATE__ && store.replaceState(window.__INITIAL_STATE__)
  7. 此时，浏览器首屏已经生成，在跳转其他路由页面就需要用到我们之前吐给浏览器的 bundle.client.js 了，跟正常的SPA应用一样，页面的路由跳转不再会走server，也就意味着不会再有SSR了

文件说明

  • app.js 用来获取vue实例，方便client和server的使用，那么server为何需要它呢？因为server需要首屏吐出页面结构，当然需要解析vue实例
  • entry-client.js 是client的打包入口，运行在浏览器环境的，所以它跟我们普通的vue项目的打包入口文件是一样的，只是多了一步替换store的动作，上面的 思路整理-6 已经说明了原因
  • entry-server.js 是server的打包入口，运行在node环境的，所以它就需要两个东西：页面结构和数据，即找到当前url对应的路由组件，并获取数据，最后将数据传入 window.__INITIAL_STATE__
